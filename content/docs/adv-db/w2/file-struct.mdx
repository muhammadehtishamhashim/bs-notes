## Before starting the topic we should know about these things:


| Term | Definition |
| :--- | :--- |
| **File organization** | The method of arranging a file of records on external storage. The record id (rid) is sufficient to physically locate a record. |
| **Index** | Data structures to organize records via trees or hashing. An index on a file speeds up selections on the search key fields for the index by trading space for speed. |
| **Search key** | Any subset of the fields of a relation can be the search key for an index on the relation. "Search key" is not the same as "key" (which is the minimal set of fields that uniquely identify a tuple). |
| **Data entry (k*)** | Pointed to by k, it contains the record id(s) of the data record or the actual record itself. |
| **Hash function** | A mapping function (h) that maps all sets of search-keys (K) to the bucket address where actual records are placed. |
| **Bucket** | Hash files store data in a bucket format. A bucket is considered a unit of storage that typically stores one complete disk block, which in turn can store one or more records. |

    What is an Index?
  </h3>
  <div className="text-sm text-muted-foreground space-y-3">
    <div>
      It is a separate small file that helps you find data in a large file (like the index at the back of a textbook).
      <ul className="list-disc pl-5 mt-2 space-y-1">
        <li><strong>Primary Index:</strong> Built on the main key of the table.</li>
        <li><strong>Secondary Index:</strong> Built on other columns (like Name or City) to speed up different types of searches.</li>
      </ul>
    </div>
    <div>
      <strong>Access Path:</strong> This is the "route" the database chooses to find your data. It decides: "Should I use the Index or just scan the whole Table?"
    </div>
  </div>
</div>

## Traditional File Structures in DBMS

<div className="my-6 rounded-xl border border-primary/20 bg-primary/5 px-5 py-3 shadow-sm">
  <h3 className="text-lg font-bold text-primary mb-2 flex items-center gap-2">
    <svg className="w-5 h-5 shrink-0" fill="none" viewBox="0 0 24 24" stroke="currentColor" strokeWidth="2"><path strokeLinecap="round" strokeLinejoin="round" d="M19 11H5m14 0a2 2 0 012 2v6a2 2 0 01-2 2H5a2 2 0 01-2-2v-6a2 2 0 012-2m14 0V9a2 2 0 00-2-2M5 11V9a2 2 0 002-2m0 0V5a2 2 0 012-2h6a2 2 0 012 2v2M7 7h10" /></svg>
    File Structures & Access Paths Concept
  </h3>
  <div className="text-sm text-muted-foreground space-y-3">
    <div>
      The database is stored as a collection of files. Each file is a sequence of records, and a record is a sequence of fields. There are many alternative file organizations, each ideal for some situations, and not so good in others. The physical database designer must determine an efficient file organization for each base relation. From the user’s viewpoint, the internal storage representation for relations should be transparent—the user should be able to access relations and tuples without having to specify where or how the tuples are stored.
    </div>
    <div>
      An access path refers to the strategy or data structure used by the DBMS to retrieve data. For example, to find the data you want, the DBMS can scan the file sequentially, use a binary search if the file is ordered, or search an index (like a B+ tree or hash index) using the search key, and then use the pointer to retrieve the data.
    </div>
  </div>
</div>
