# Timestamp Ordering Protocol

Locking protocols have a major downside: they force transactions to wait, which can lead to Deadlocks. **Timestamp Ordering** is an alternative, optimistic approach that uses no locks at all. 

Instead of locking data, the DBMS assigns a unique, chronological timestamp to every transaction when it starts. The protocol then forces transactions to execute logically in the order of their timestamps.

### How it Works

1.  **Transaction Timestamp ($TS$):** When Transaction 1 starts, it gets $TS(T_1) = 1$. When Transaction 2 starts later, it gets $TS(T_2) = 2$.
    * *Older transaction:* Smaller timestamp.
    * *Younger transaction:* Larger timestamp.

2.  **Data Item Timestamps:** Every piece of data (e.g., Row X) in the database records two timestamps:
    * **Read Timestamp $RTS(X)$:** The timestamp of the youngest transaction that successfully *read* the data.
    * **Write Timestamp $WTS(X)$:** The timestamp of the youngest transaction that successfully *wrote/updated* the data.

---

## The Read and Write Rules

When a transaction $T_i$ tries to Read or Write a piece of data ($X$), the DBMS checks the timestamps. The core logic is: **"You cannot access data from the future."**

### 1. The Read Rule
If transaction $T_i$ wants to **READ** $X$:
* If $TS(T_i) < WTS(X)$: This means a *younger* transaction has already overwritten this data. $T_i$ is trying to read data from its own "future". 
    * **Action:** $T_i$ is rejected, aborted, and restarted with a new timestamp.
* If $TS(T_i) \geq WTS(X)$: Safe! The data was written before $T_i$ started.
    * **Action:** Allow the read. Update $RTS(X)$ to $TS(T_i)$ (if $T_i$'s timestamp is larger).

### 2. The Write Rule
If transaction $T_i$ wants to **WRITE** $X$:
* If $TS(T_i) < RTS(X)$: A younger transaction has already read the old value. If we write now, we ruin their read.
    * **Action:** $T_i$ is rejected, aborted, and restarted.
* If $TS(T_i) < WTS(X)$: A younger transaction has already overwritten this data.
    * **Action:** Usually rejected and rolled back. *(See Thomas Write Rule below).*
* Otherwise: Safe!
    * **Action:** Allow the write. Update $WTS(X)$ to $TS(T_i)$.

<Callout title="Thomas Write Rule" type="warn">
There is a special exception to the Write Rule called the **Thomas Write Rule**. 

If $T_i$ tries to write, but $TS(T_i) < WTS(X)$, it means a younger transaction has already written over this data. Instead of completely aborting the older $T_i$, the Thomas Write Rule says: **Just ignore $T_i$'s write operation and let it continue.** Since the younger transaction's write is the most "recent" in logical time, $T_i$'s write is obsolete anyway.
</Callout>

## Comprehensive Summary

* **Timestamp Ordering:** A non-locking protocol. Transactions are given logical timestamps at startup to determine their execution priority.
* **Data Timestamps:** Every row tracks the timestamp of the last transaction to Read ($RTS$) and Write ($WTS$) it.
* **The Golden Rule:** A transaction cannot access data that has already been read or written by a transaction with a *larger* (younger/future) timestamp. If it tries, it is aborted and restarted with a new, current timestamp.
* **Thomas Write Rule:** If an older transaction tries to write to data already overwritten by a younger transaction, the older write is simply ignored instead of aborting the transaction.
