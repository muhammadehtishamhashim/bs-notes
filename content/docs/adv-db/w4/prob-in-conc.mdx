# Concurrency in Databases

**Concurrency** means allowing multiple users or transactions to access and modify the database at the exact same time. While this drastically improves performance and system utilization, uncontrolled interleaving of operations can destroy database consistency.

There are three major problems caused by uncontrolled concurrency, along with the issue of Deadlocks (which arises when trying to control concurrency).

---

## 1. The Lost Update Problem

A successfully completed update operation by one user is completely overwritten and "lost" by another user doing a simultaneous update.

**Example Scenario:**
Transaction 1 ($T_1$) withdraws £10 from an account. Transaction 2 ($T_2$) deposits £100 into the same account. The initial balance is £100.

## 2. Uncommitted Dependency (Dirty Read)

This problem occurs when a transaction is allowed to read the intermediate, uncommitted results of another transaction. If the first transaction fails and rolls back, the second transaction has read "dirty" (invalid) data.

**Example Scenario:**
$T_4$ updates a balance from £100 to £200, but later encounters an error and aborts. Meanwhile, $T_5$ reads that £200 balance to do calculations.

## 3. Deadlocks

While controlling concurrency (usually via locks), we can accidentally create a situation where two or more transactions are waiting indefinitely for each other to release resources. This is called a **Deadlock** or Deadly Embrace.

**Example Scenario:**
$T_1$ locks Item A and needs Item B to finish.
$T_2$ locks Item B and needs Item A to finish.

## Comprehensive Summary

* **Concurrency:** Multiple transactions executing simultaneously. Necessary for speed, but dangerous if uncontrolled.
* **Lost Update:** Two transactions read the same data, calculate updates in RAM, and write back. The first write is overwritten and lost.
* **Dirty Read (Uncommitted Dependency):** A transaction reads data written by another transaction *before* it commits. If the writer rolls back, the reader is using ghost data.
* **Deadlocks:** A side-effect of using locks to fix the above problems. Two transactions hold locks the other needs, resulting in an infinite standoff.
