# Concept of Transactions

A **transaction** is a single logical unit of work in a Database Management System (DBMS). It consists of an action or a series of actions—carried out by a single user or application program—that accesses or changes the contents of the database.

A transaction could be an entire program, a function, or just a single SQL statement (like `INSERT` or `UPDATE`). The primary goal of a transaction is to transform the database from **one consistent state to another**. 

Because the DBMS doesn't inherently know which updates are grouped together, we define transaction boundaries using explicit commands:
* `BEGIN TRANSACTION`: Marks the start of the logical unit.
* `COMMIT`: Successfully ends the transaction, saving all changes permanently to the disk.
* `ROLLBACK`: Aborts the transaction, undoing any changes made in RAM and reverting to the previous consistent state.

<Callout title="Two Ultimate Outcomes" type="info">
A transaction can only end in one of two ways:
1. **Committed:** Execution was 100% successful. The database reaches a new consistent state, and changes are irreversible (permanent).
2. **Aborted (Rolled Back):** Execution failed midway (e.g., due to a crash or rule violation). The database is undone and restored to the exact consistent state it was in *before* the transaction started.
</Callout>

## Comprehensive Summary

* **Transaction:** A logical unit of work (group of DB operations) aiming to take the database from one valid state to another.
* **Boundaries:** Defined by `BEGIN TRANSACTION`, `COMMIT`, and `ROLLBACK`.
* **Flow:** Data is Read from DB $\rightarrow$ Modified in RAM $\rightarrow$ Written to RAM $\rightarrow$ Committed to permanent Hard Disk storage.
* **ACID Properties:**
  * **Atomicity:** Entire transaction executes fully, or not at all.
  * **Consistency:** Data validity is maintained before and after the transaction.
  * **Isolation:** Concurrent transactions don't interfere with one another.
  * **Durability:** Committed changes survive system failures.
