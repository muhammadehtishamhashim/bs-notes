# The Assembly Phase (`as`)

The assembler is a relatively straightforward program compared to the compiler. Its primary job is to take the textual assembly instructions generated by the compiler and translate them into pure **machine code** (binary 0s and 1s).

### Understanding the Relocatable Object File

The output of this phase is an `.o` file (e.g., `hello.o`). This is called a **Relocatable Object File**. 

* **Binary Format:** You can no longer open this file in a normal text editor. It contains binary data structured in a specific format (like ELF on Linux or PE on Windows).
* **Why "Relocatable"?** The code is compiled into blocks, but it doesn't know *exactly* where in the physical RAM it will be loaded yet. So, memory addresses for variables and functions are left as relative offsets (placeholders), waiting to be "relocated" by the linker.
* **Symbol Table:** The `.o` file includes a dictionary of all functions and global variables defined in the file, and a list of functions it needs but doesn't have (like `printf`).

## Comprehensive Summary
* **What it is:** The translator from text-based CPU mnemonics to pure hardware binary.
* **What it does:** Reads assembly instructions and outputs their exact binary equivalents (opcodes). It packages this binary data into a structured object file.
* **Why it matters:** CPUs do not read text. They only process binary voltages. This phase creates the actual instructions the CPU will execute.
* **The Result:** A `.o` relocatable object file. It contains valid machine code, but it is "incomplete" because external library dependencies haven't been linked yet.
